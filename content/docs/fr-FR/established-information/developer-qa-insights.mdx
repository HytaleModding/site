# Aperçus techniques des Q&R avec les développeurs

Cette session de questions-réponses approfondie avec les développeurs de Hytale révèle des détails techniques essentiels sur l'architecture du jeu, la philosophie de modding, les systèmes de gameplay et l'approche de développement.

## Architecture technique

### Réseau et infrastructure

* **Protocole** : Utilise QUIC (UDP fiable avec canaux non fiables) au lieu de TCP
* **Compatibilité serveur** : Fonctionne sur toute plateforme supportant Java 25
* **Méthode de connexion** : IP directe initialement, découverte de serveurs prévue après le lancement
* **Couche de données** : Le format de sérialisation personnalisé actuel sera remplacé après le lancement
* **Performance** : Support multi-cœur - chaque monde a un thread principal + exécution parallèle
* **Rendu** : OpenGL 3.3 pour la compatibilité Mac, migration potentielle future vers Vulkan/Metal
* **Modèle de tick** : Le taux de tick du serveur de base est de 30 TPS par défaut ; peut être modifié via des plugins serveur.

### Sécurité et distribution

* **Pas d'exécution de code côté client** : Empêche les exploits d'exécution de code à distance
* **Téléchargements limités à** : Fichiers de configuration et assets uniquement
* **Distribution des mods** : Plateformes communautaires (CurseForge, Modrinth), pas de marketplace en jeu immédiate
* **Hébergement de serveur** : Disponible dès le jour 1, aucun accord commercial requis (juste EULA/CGU)

## Capacités de modding

### Puissance côté serveur

* **Accès Java complet** : Connexions bases de données, requêtes web, toute fonctionnalité Java
* **Bibliothèques personnalisées** : Peut créer du scripting Lua, des frameworks ML, ou toute bibliothèque Java
* **Modification du serveur** : Le code source partagé permet des modifications lourdes du serveur
* **Hot Reload** : Supporté pour la plupart des types d'assets
* **Documentation** : GitBook public prévu, bien qu'initialement épars

### Création de contenu (sans code requis)

* **Génération de monde** : Biomes et cartes personnalisés sans Java
* **PNJ et objets** : Configuration basée sur JSON
* **Recettes de craft** : Système piloté par les données
* **Scripting visuel** : Remplace les blocs de commande, entièrement sandboxé
* **Système de prefabs** : Blocs de construction sophistiqués avec éditeur en jeu

### Modding avancé

* **Système de combat** : Système d'interaction piloté par les données pour les chaînes d'attaques et les capacités
* **Sous-hitboxes** : Support pour les mécaniques de boss complexes (nécessite des améliorations)
* **Effets de statut** : Système de buff/debuff avec effets et stats personnalisés
* **UI personnalisée** : La transition vers NoesisGUI permet une personnalisation puissante de l'interface
* **Physique des blocs** : Système sélectif (les arbres tombent, mais les blocs placés ne tombent pas)

## Systèmes de gameplay

### Mécaniques de base

* **Système d'endurance** : Pas de faim par défaut (peut être moddé)
* **Gravité** : Physique de blocs sélective - les arbres s'effondrent, les bâtiments peuvent flotter
* **Éclairage** : Refonte majeure pour corriger les incohérences modèle/voxel
* **Hauteur du monde** : Actuellement limitée, peut changer avec les mises à jour du stockage de voxels
* **Automatisation** : Pas d'équivalent redstone au lancement (moddable)

### Systèmes de joueur

* **Modèles de personnage** : Humains par défaut, modifiables en mode créatif
* **Avatars multijoueur** : Contrôlés par le serveur (pas de modèles personnalisés en multijoueur)
* **Skins** : Personnalisation complète possible (« transformez-vous en patate »)
* **Métamorphose** : Des potions de transformation existent pour le fun/modding

### Monde et PNJ

* **Système de factions** : Monde vivant avec interactions PNJ dynamiques
* **Systèmes de villages** : Villages dynamiques prévus avec relations et comportements
* **Chargement des chunks** : Les entités se déchargent quand elles sont hors de la distance de rendu de tous les joueurs
* **Seeds** : Seeds de génération de monde partageables supportés

## Philosophie de développement

### Stratégie de sortie

* **Priorité au jeu de base** : Le mode aventure est l'objectif principal, l'architecture de modding le supporte
* **Dette technique** : Reconnue comme nécessaire pour la sortie, engagement à l'amélioration
* **Fréquence des mises à jour** : Mises à jour très fréquentes prévues, surtout après le lancement
* **Politique de version** : Toujours la dernière version, mais les serveurs peuvent exécuter des versions modifiées

### Focus communauté

* **Support aux créateurs** : Primes de développement prévues pour les contributeurs communautaires
* **Tutoriels** : Tutoriels de modding officiels prévus avec des partenaires
* **Intégration des retours** : Fort engagement envers les améliorations pilotées par la communauté
* **Accessibilité** : Conçu pour le matériel bas de gamme, exigences système raisonnables

### Outils et documentation

* **Intégration Blockbench** : Remplace le « Hytale Model Maker » personnalisé
* **Outils Machinima** : Capacités complètes de création cinématique
* **Outils créatifs** : Révélation majeure prévue pour les fonctionnalités de construction/création
* **Accès au code source** : Le code serveur sera non-obfusqué avec les commentaires des développeurs

## Support des plateformes

* **Actuel** : Windows (principal)
* **Prévu** : Support Mac/Linux activement en développement
* **VR** : Non faisable dans un futur proche
* **Performance** : Framerate non plafonné, conçu pour l'accessibilité
* **Jeu hors ligne** : Le solo fonctionne sans connexion internet

## Monétisation et légal

* **Pas de Pay-to-Win** : Engagement ferme contre les mécaniques P2W
* **Cosmétiques uniquement** : Packs cosmétiques à bas prix pour soutenir l'infrastructure
* **EULA Serveur** : Conditions en cours de finalisation pour les politiques de monétisation des serveurs
* **Blocage QUIC** : Problème reconnu dans certaines régions, pas de protocole alternatif

## Limitations actuelles

* **Shaders personnalisés** : Non supportés jusqu'à ce qu'une solution de sandboxing soit trouvée
* **Limitations UI/Input** : En cours d'extension mais actuellement restreintes
* **Documentation** : Éparse initialement, s'améliorera avec le temps
* **Mods client** : Pas d'exécution de code côté client pour la sécurité
* **Nameplates** : Personnalisation actuellement limitée

Cette base technique complète positionne Hytale comme une plateforme sécurisée et extensible qui donne du pouvoir aux créateurs tout en maintenant la performance, l'accessibilité et un engagement à l'amélioration continue.
