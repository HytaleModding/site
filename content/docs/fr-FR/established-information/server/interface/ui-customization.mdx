L'interface peut être personnalisée via le modding **côté serveur**. De nombreuses pages d'interface affichées aux joueurs sont disponibles pour être modifiées par les propriétaires de serveurs via des mods.

## Personnalisation de l'interface dans Hytale

Hytale fournit des capacités de personnalisation d'interface étendues via le modding côté serveur, permettant aux propriétaires de serveurs de modifier de nombreuses pages d'interface avec lesquelles les joueurs interagissent.

### Flux de contrôle

Le système d'interface dans Hytale suit un schéma de flux de contrôle spécifique :

1. Le serveur envoie les données d'interface au client
2. Le client affiche l'interface
3. L'utilisateur interagit avec l'interface (par ex. en appuyant sur un bouton)
4. Le client envoie l'événement d'interaction au serveur
5. Le serveur exécute la logique de callback pour l'événement

Cette architecture signifie :

* Le client est responsable de l'affichage de l'interface et de l'envoi des interactions utilisateur au serveur
* Le serveur gère toute la logique réelle derrière les interactions utilisateur

<Callout type="warning" title="Implications possibles sur les performances">
  <span>
    Parce que le serveur traite toute la logique d'interface, il y a un impact
    de latence inhérent sur la réactivité de l'interface. Les clics de bouton et
    autres interactions doivent voyager jusqu'au serveur avant de prendre effet,
    ce qui signifie que le ping réseau affecte directement la réactivité de
    l'interface.
  </span>

  <span>
    De plus, puisque le modding côté client n'est pas supporté, aucune valeur
    moddée ne persiste sur le client.
  </span>
</Callout>

### Assets de mise en page d'interface

Le système de personnalisation d'interface de Hytale fonctionne via la structure suivante :

* **Mise en page d'interface** : Envoyée au client comme un asset
* **Comportement d'interface** : Défini dans du code Java
* **Mises en page personnalisées** : Permet la création de mises en page d'interface sur mesure plutôt que de réutiliser des mises en page existantes (contrairement aux modifications de menu d'inventaire dans Minecraft)

### Implémentations du framework d'interface

Hytale supporte actuellement plusieurs implémentations d'interface, avec des plans pour se consolider sur NoesisGUI :

#### Implémentation legacy

* Sert d'exemple des capacités de modding serveur
* Utilise une combinaison d'éléments en jeu, d'assets de balisage d'interface et de code de mod Java
* Utilise un langage de balisage personnalisé legacy
* Sera déprécié en faveur de NoesisGUI

#### NoesisGUI (Framework principal)

* Utilise le langage de balisage XAML pour la définition d'interface
* Supporte plusieurs langues pour l'internationalisation
* Fonctionne sur des bindings C# côté client (pas C++)
* Représente la direction future du système d'interface de Hytale

#### Troisième framework

* Actuellement en cours de suppression progressive
* Sera remplacé par NoesisGUI dans le cadre de l'effort de consolidation

### Considérations de performance

En raison de la nature côté serveur du traitement d'interface, il y a des caractéristiques de performance inhérentes à considérer :

* **Impact de latence** : Les clics de bouton et les interactions d'interface doivent voyager jusqu'au serveur avant de prendre effet, ce qui signifie que le ping réseau affecte directement la réactivité de l'interface
* **Pas de persistance client** : Puisque le modding côté client n'est pas supporté, aucune valeur moddée ne persiste sur le client
* **Architecture optimisée** : Le système de Hytale est conçu pour minimiser la latence via des protocoles de communication efficaces et des performances serveur optimisées, garantissant des interactions fluides malgré les exigences de traitement côté serveur

L'architecture priorise la sécurité et la cohérence en centralisant la logique d'interface sur le serveur tout en maintenant des expériences utilisateur réactives.
