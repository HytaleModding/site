Os sistemas são onde a lógica reside. Enquanto os componentes são contêineres puros de dados, os sistemas operam sobre entidades que correspondem a consultas específicas de componentes. O agendador do ECS executa os sistemas a cada tick, passando apenas as entidades que têm os componentes relevantes para o sistema.

### `EntityTickingSystem`

O tipo de sistema mais comum. Ele é executado por tick e processa individualmente cada entidade que corresponde à sua query.

```java
public class PoisonSystem extends EntityTickingSystem<EntityStore> {

  private final ComponentType<EntityStore, PoisonComponent> poisonComponentType;

  public PoisonSystem(ComponentType<EntityStore, PoisonComponent> poisonComponentType) {
    this.poisonComponentType = poisonComponentType;
  }

  @Override
  public void tick(float dt, int index, @Nonnull ArchetypeChunk<EntityStore> archetypeChunk,
      @Nonnull Store<EntityStore> store, @Nonnull CommandBuffer<EntityStore> commandBuffer) {

    PoisonComponent poison = archetypeChunk.getComponent(index, poisonComponentType);
    Ref<EntityStore> ref = archetypeChunk.getReferenceTo(index);

    poison.addElapsedTime(dt);

    if (poison.getElapsedTime() >= poison.getTickInterval()) {
      poison.resetElapsedTime();

      Damage damage = new Damage(Damage.NULL_SOURCE, DamageCause.OUT_OF_WORLD, poison.getDamagePerTick());
      DamageSystems.executeDamage(ref, commandBuffer, damage);

      poison.decrementRemainingTicks();
    }

    if (poison.isExpired()) {
      commandBuffer.removeComponent(ref, poisonComponentType);
    }
  }

  @Nullable
  @Override
  public SystemGroup<EntityStore> getGroup() {
    return DamageModule.get().getGatherDamageGroup();
  }

  @Nonnull
  @Override
  public Query<EntityStore> getQuery() {
    return Query.and(this.poisonComponentType);
  }
}
```

O método `tick` recebe `dt` que é o delta de tempo desde o último tick. Isso permite acumular tempo para lógicas baseadas em intervalo, em vez de simplesmente contar ticks. O `ArchetypeChunk` fornece acesso aos componentes da entidade por meio do índice e `getReferenceTo` retorna o `Ref` necessário para emitir comandos.

### `TickingSystem`

É executado uma vez por tick de forma global, não por entidade. Use esse tipo para atualizações globais do mundo ou lógicas que não têm como alvo entidades específicas.

```java
public class GlobalUpdateSystem extends TickingSystem<EntityStore> {

  @Override
  public void tick(float dt, int index, Store<EntityStore> store) {
    World world = store.getExternalData().getWorld();
  }
}
```

### `DelayedEntitySystem`

Semelhante ao `EntityTickingSystem`, mas com um atraso embutido. O construtor recebe um `float` que representa os segundos entre execuções.

```java
public class HealthRegenSystem extends DelayedEntitySystem<EntityStore> {

  public HealthRegenSystem() {
    super(1.0f);
  }

  @Override
  public void tick(float dt, int index, @Nonnull ArchetypeChunk<EntityStore> archetypeChunk,
      @Nonnull Store<EntityStore> store, @Nonnull CommandBuffer<EntityStore> commandBuffer) {
    // Executa a cada 1 segundo por entidade correspondente
  }

  @Nonnull
  @Override
  public Query<EntityStore> getQuery() {
    return Query.and(Player.getComponentType());
  }
}
```

### `RefSystem`

Também podemos criar sistemas para escutar mudanças nas próprias entidades. Isso é feito por meio de um `RefSystem`. Por exemplo, se quisermos executar uma ação sempre que um componente for adicionado, atualizado ou removido de uma entidade.
Isso pode ser feito usando um `RefChangeSystem<ECS_STORE, COMPONENT>`. Vamos detalhar o exemplo abaixo:

```java
public class PermissionAttachmentSystem extends RefChangeSystem<EntityStore, PermissionAttachment> {

  private static final HytaleLogger LOGGER = HytaleLogger.forEnclosingClass();

  @Nonnull
  @Override
  public ComponentType<EntityStore, PermissionAttachment> componentType() {
    return EntityStoreRegistry.get().getPermissionAttachmentComponentType();
  }

  @Override
  public void onComponentAdded(@Nonnull Ref<EntityStore> ref, @Nonnull PermissionAttachment permissionAttachment, @Nonnull Store<EntityStore> store, @Nonnull CommandBuffer<EntityStore> commandBuffer) {
    UUIDComponent component = store.getComponent(ref, UUIDComponent.getComponentType());
    UUID playerUuid = component.getUuid();

    // O componente PermissionAttachment foi adicionado a uma nova entidade
  }

  @Override
  public void onComponentSet(@Nonnull Ref<EntityStore> ref, @Nullable PermissionAttachment oldAttachment, @Nonnull PermissionAttachment newAttachment, @Nonnull Store<EntityStore> store, @Nonnull CommandBuffer<EntityStore> commandBuffer) {
    UUIDComponent component = store.getComponent(ref, UUIDComponent.getComponentType());
    UUID playerUuid = component.getUuid();

    // O componente PermissionAttachment foi atualizado usando replaceComponent ou putComponent
  }

  @Override
  public void onComponentRemoved(@Nonnull Ref<EntityStore> ref, @Nonnull PermissionAttachment permissionAttachment, @Nonnull Store<EntityStore> store, @Nonnull CommandBuffer<EntityStore> commandBuffer) {
    UUIDComponent component = store.getComponent(ref, UUIDComponent.getComponentType());
    UUID playerUuid = component.getUuid();

    // O componente PermissionAttachment foi removido de uma entidade
  }

  @Nullable
  @Override
  public Query<EntityStore> getQuery() {
    return EntityStoreRegistry.get().getPermissionAttachmentComponentType();
  }
}
```

Nesse sistema, escutamos mudanças no componente `PermissionAttachment` dentro do `EntityStore`. Com isso, temos acesso aos métodos `onComponentAdded`, `onComponentSet` e `onComponentRemoved` que nesse exemplo fazem cache e persistem dados de permissão armazenados em um componente `PermissionAttachment` personalizado associado a um jogador.

## `Queries`

As `queries` filtram quais entidades um sistema processa. O ECS só passa entidades para o método `tick` do seu sistema se elas tiverem todos os componentes especificados na `query`.

```java
// Componente único — qualquer entidade com PoisonComponent
Query.and(poisonComponentType)

// Múltiplos componentes — entidades com ambos
Query.and(poisonComponentType, Player.getComponentType())

// Exclusão — jogadores que não estão mortos
Query.and(Player.getComponentType(), Query.not(DeathComponent.getComponentType()))
```

## `SystemGroups` e `Dependencies`

Os sistemas podem declarar a qual grupo pertencem e quais dependências eles têm incluídas. Isso controla a ordem de execução, o que é fundamental para sistemas que interagem entre si.

```java
@Nullable
@Override
public SystemGroup<EntityStore> getGroup() {
  return DamageModule.get().getGatherDamageGroup();
}
```

Para ordenações mais complexas, você pode sobrescrever `getDependencies`:

```java
@Nonnull
public Set<Dependency<EntityStore>> getDependencies() {
  return Set.of(
    new SystemGroupDependency(Order.AFTER, DamageModule.get().getFilterDamageGroup()),
    new SystemDependency(Order.BEFORE, PlayerSystems.ProcessPlayerInput.class)
  );
}
```

O sistema de dano é um bom exemplo que mostra a importância da ordem. O pipeline de dano do Hytale tem quatro estágios: `GatherDamageGroup` coleta as fontes de dano; `FilterDamageGroup` aplica reduções e cancelamentos; depois o dano é aplicado à vida; e, por fim, `InspectDamageGroup` trata efeitos colaterais como partículas e sons. Se esses estágios fossem executados na ordem errada, animações de morte poderiam tocar antes da entidade morrer ou a redução de armadura poderia ser aplicada depois que a vida já tivesse sido subtraída.

## Registrando `components` e `systems`

Os `Components` e `systems` devem ser registrados ao configurar o plugin. O `EntityStoreRegistry` é responsável por isso.

```java
public final class ExamplePlugin extends JavaPlugin {

  private static ExamplePlugin instance;
  private ComponentType<EntityStore, PoisonComponent> poisonComponent;

  public ExamplePlugin(@Nonnull JavaPluginInit init) {
    super(init);
    instance = this;
  }

  @Override
  protected void setup() {
    this.getCommandRegistry().registerCommand(new ExampleCommand());
    this.getEventRegistry().registerGlobal(PlayerReadyEvent.class, ExampleEvent::onPlayerReady);
    this.getEventRegistry().registerGlobal(PlayerChatEvent.class, ChatFormatter::onPlayerChat);

    this.poisonComponent = this.getEntityStoreRegistry()
        .registerComponent(PoisonComponent.class, PoisonComponent::new);
    this.getEntityStoreRegistry().registerSystem(new PoisonSystem(this.poisonComponent));
  }

  public ComponentType<EntityStore, PoisonComponent> getPoisonComponentType() {
    return poisonComponent;
  }

  public static ExamplePlugin get() {
    return instance;
  }
}
```

O método `registerComponent` retorna um `ComponentType`, ele atua como uma chave para acessar esse tipo de componente em todo o plugin. Armazene-o como um campo e o passe para quaisquer sistemas que precisem dele. O segundo argumento é uma fábrica para criar instâncias padrão.
