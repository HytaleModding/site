In this guide, you'll learn how to manage permission nodes and groups in your Hytale plugin. Permissions control what actions players can perform on your server, from basic commands to advanced administrative functions.

## O que são permissões?

Permissions are string identifiers that control access to features and commands. When a player tries to perform an action (like using a command), the server checks if they have the required permission.

**Example permission nodes:**

- `hytale.command.gamemode` – Permissão para usar o comando de modo de jogo;
- `hytale.editor.brush.use` – Permissão para usar ferramentas de pincel;
- `myplugin.teleport.home` – Permissão personalizada para seu plugin;

As permissões seguem um padrão hierárquico de nomenclatura: `namespace.category.action`

## Principais conceitos

Antes de partir para o código, vamos entender os conceitos centrais:

| Conceito            | Descrição                                                                        |
| ------------------- | -------------------------------------------------------------------------------- |
| **Permission Node** | Uma _string_ como `myplugin.feature.use` que representa uma permissão específica |
| **Group**           | Um conjunto nomeado de permissões (ex.: `Admin`, `VIP`, `Default`)               |
| **Provider**        | O backend responsável por armazenar e recuperar permissões                       |
| **Wildcard**        | Um padrão como `*` ou `myplugin.*` que corresponde a várias permissões           |

### Como as permissões são verificadas

Ao verificar se um jogador possui uma permissão, o Hytale segue a seguinte ordem:

1. **User's direct permissions** - Permissions granted directly to the player
2. **Group permissions** - Permissions from groups the player belongs to
3. **Virtual groups** - Game-mode-based permissions (e.g., Creative mode grants builder tools)
4. **Default value** - Falls back to `false` if no match found

<Callout type="info">
The first definitive match wins. If a player has a permission granted at the user level, group permissions won't override it.
</Callout>

## The `PermissionsModule` Class

The `PermissionsModule` class is the central hub for managing permissions. Access it using the singleton pattern:

```java
PermissionsModule perms = PermissionsModule.get();
```

### Available Operations

The module supports these operations:

| Operação                         | Método                                          |
| -------------------------------- | ----------------------------------------------- |
| Adiciona permissões a um usuário | `addUserPermission(uuid, permissions)`          |
| Remove permissões de um usuário  | `removeUserPermission(uuid, permissions)`       |
| Adiciona permissões a um grupo   | `addGroupPermission(groupName, permissions)`    |
| Remove permissões de um grupo    | `removeGroupPermission(groupName, permissions)` |
| Adiciona um usuário a um grupo   | `addUserToGroup(uuid, groupName)`               |
| Remove um usuário de um grupo    | `removeUserFromGroup(uuid, groupName)`          |
| Obtém grupos de usuários         | `getGroupsForUser(uuid)`                        |
| Verifica permissão               | `hasPermission(uuid, permissionNode)`           |

<Callout type="warning">
The module does **not** support:
- Listing all defined groups
- Deleting a group entirely

Groups are implicitly created when you first add permissions to them.
</Callout>

## Checking Permissions

The most common operation is checking if a player has a permission.

### Basic Permission Check

```java
// Returns true if player has the permission, false otherwise
boolean canUse = PermissionsModule.get().hasPermission(playerUUID, "myplugin.feature.use");

if (canUse) {
    // Player has permission, allow the action
} else {
    // Player lacks permission, deny or show error
}
```

### Permission Check with Default Value

You can specify what to return when no permission is explicitly set:

```java
// Returns true by default if no explicit permission is found
boolean canUse = PermissionsModule.get().hasPermission(playerUUID, "myplugin.feature.use", true);
```

This is useful for features that should be enabled by default but can be revoked.

### Using PermissionHolder Interface

Players and command senders implement `PermissionHolder`, allowing direct checks:

```java
// In a command or event handler where you have access to the player
if (player.hasPermission("myplugin.admin.manage")) {
    // Player has admin permission
}
```

## Managing User Permissions

### Adding Permissions to a User

Permissions are additive - new permissions are added on top of existing ones, not replaced.

```java
PermissionsModule perms = PermissionsModule.get();

// Add a single permission
perms.addUserPermission(playerUUID, Set.of("myplugin.vip.chat"));

// Add multiple permissions at once
Set<String> newPerms = Set.of(
    "myplugin.vip.chat",
    "myplugin.vip.fly",
    "myplugin.vip.kit"
);
perms.addUserPermission(playerUUID, newPerms);
```

### Removing Permissions from a User

```java
PermissionsModule perms = PermissionsModule.get();

// Remove specific permissions
Set<String> toRemove = Set.of("myplugin.vip.fly");
perms.removeUserPermission(playerUUID, toRemove);
```

## Managing Groups

Groups let you assign a collection of permissions to multiple players. This is more manageable than assigning permissions individually.

### Adding Permissions to a Group

If the group doesn't exist, it will be created automatically.

```java
PermissionsModule perms = PermissionsModule.get();

// Create/update the "VIP" group with permissions
Set<String> vipPerms = Set.of(
    "myplugin.vip.chat",
    "myplugin.vip.fly",
    "myplugin.vip.kit"
);
perms.addGroupPermission("VIP", vipPerms);
```

### Removing Permissions from a Group

```java
PermissionsModule perms = PermissionsModule.get();

// Remove specific permissions from the group
Set<String> toRemove = Set.of("myplugin.vip.fly");
perms.removeGroupPermission("VIP", toRemove);
```

### Adding a User to a Group

```java
PermissionsModule perms = PermissionsModule.get();

// Add player to the VIP group - they now have all VIP permissions
perms.addUserToGroup(playerUUID, "VIP");
```

### Removing a User from a Group

```java
PermissionsModule perms = PermissionsModule.get();

// Remove player from VIP group
perms.removeUserFromGroup(playerUUID, "VIP");
```

### Getting a User's Groups

```java
PermissionsModule perms = PermissionsModule.get();

// Get all groups the player belongs to
Set<String> groups = perms.getGroupsForUser(playerUUID);

for (String group : groups) {
    System.out.println("Player is in group: " + group);
}
```

## Grupos integrados

O Hytale inclui dois grupos por padrão:

| Grupo     | Permissões (padrão)       | Finalidade                              |
| --------- | ------------------------- | --------------------------------------- |
| `OP`      | `*` (todas as permissões) | Operadores do servidor com acesso total |
| `Default` | Nenhuma                   | Grupo base para todos os jogadores      |

Os jogadores sem atribuição explícita de grupo fazem parte automaticamente do grupo `Default`.

## Wildcards

Os Wildcards permitem conceder ou negar várias permissões usando uma única expressão.

### Expressões de Wildcard

| Padrão              | Significado                                             |
| ------------------- | ------------------------------------------------------- |
| `*`                 | Concede **todas** as permissões                         |
| `myplugin.*`        | Concede todas as permissões que começam com `myplugin.` |
| `-*`                | Nega **todas** as permissões                            |
| `-myplugin.admin.*` | Nega todas as permissões administrativas do seu plugin  |

### Exemplo: grupo administrativo com wildcards

```java
// Concede todas as permissões aos administradores
perms.addGroupPermission("Admin", Set.of("*"));

// Concede todas as permissões do plugin aos moderadores
perms.addGroupPermission("Moderator", Set.of("myplugin.*"));

// Concede todas as permissões, exceto comandos administrativos
perms.addGroupPermission("Helper", Set.of(
    "*",
    "-myplugin.admin.*"  // Nega permissões administrativas
));
```

<Callout type="warning">
As permissões de negação (iniciadas com `-`) têm precedência em cada nível. Use com cuidado para evitar bloqueios acidentais.
</Callout>

## Nós de permissão integrados

O Hytale fornece constantes para permissões comuns por meio da classe `HytalePermissions`.

### Permissões de comandos

```java
// Gera a permissão para um comando
String perm = HytalePermissions.fromCommand("gamemode");
// Resultado: "hytale.command.gamemode"

// Gera a permissão para um subcomando
String perm = HytalePermissions.fromCommand("gamemode", "creative");
// Resultado: "hytale.command.gamemode.creative"
```

### Nós de permissão comuns

| Permissão                     | Descrição                      |
| ----------------------------- | ------------------------------ |
| `hytale.command.op.add`       | Adiciona jogadores ao grupo OP |
| `hytale.command.op.remove`    | Remove jogadores do grupo OP   |
| `hytale.editor.brush.use`     | Usa as ferramentas de pincel   |
| `hytale.editor.prefab.use`    | Usa os prefabs                 |
| `hytale.editor.selection.use` | Usa as ferramentas de seleção  |
| `hytale.editor.history`       | Desfaz/refaz operações         |
| `hytale.camera.flycam`        | Usa o modo de câmera livre     |

## Monitorando eventos de permissão

Você pode reagir a mudanças de permissão usando o sistema de eventos do Hytale.

### Eventos disponíveis

| Evento                                           | Acionado ao                       |
| ------------------------------------------------ | --------------------------------- |
| `PlayerPermissionChangeEvent.PermissionsAdded`   | Adicionar permissões a um usuário |
| `PlayerPermissionChangeEvent.PermissionsRemoved` | Remover permissões de um usuário  |
| `PlayerGroupEvent.Added`                         | Adicionar usuário a um grupo      |
| `PlayerGroupEvent.Removed`                       | Remover usuário de um grupo       |
| `GroupPermissionChangeEvent.Added`               | Adicionar permissões a um grupo   |
| `GroupPermissionChangeEvent.Removed`             | Remover permissões de um grupo    |

### Example: Reacting to Permission Changes

```java
public class PermissionListener {

    public static void onPermissionsAdded(PlayerPermissionChangeEvent.PermissionsAdded event) {
        UUID playerUUID = event.getPlayerUuid();
        Set<String> added = event.getAddedPermissions();

        System.out.println("Permissions added to " + playerUUID + ": " + added);

        // Maybe notify the player or update cached data
    }

    public static void onGroupAdded(PlayerGroupEvent.Added event) {
        UUID playerUUID = event.getPlayerUuid();
        String groupName = event.getGroupName();

        System.out.println("Player " + playerUUID + " joined group: " + groupName);
    }
}
```

### Registrando os eventos

```java
public class MyPlugin extends JavaPlugin {

    @Override
    public void setup() {
        EventRegistry events = this.getEventRegistry();

        events.registerGlobal(
            PlayerPermissionChangeEvent.PermissionsAdded.class,
            PermissionListener::onPermissionsAdded
        );

        events.registerGlobal(
            PlayerGroupEvent.Added.class,
            PermissionListener::onGroupAdded
        );
    }
}
```

## Creating a Custom Permission Provider

For advanced use cases, you can create your own permission provider. This is useful for:

- Database-backed permissions (MySQL, MongoDB, etc.)
- Cross-server permission synchronization
- Custom features like permission expiry or contexts

### Implementando `PermissionProvider`

```java
public class DatabasePermissionProvider implements PermissionProvider {

    @Nonnull
    @Override
    public String getName() {
        return "DatabasePermissionProvider";
    }

    // User permission methods
    @Override
    public void addUserPermissions(@Nonnull UUID uuid, @Nonnull Set<String> permissions) {
        // Save to your database
    }

    @Override
    public void removeUserPermissions(@Nonnull UUID uuid, @Nonnull Set<String> permissions) {
        // Remove from your database
    }

    @Override
    public Set<String> getUserPermissions(@Nonnull UUID uuid) {
        // Query your database
        return Set.of();
    }

    // Group permission methods
    @Override
    public void addGroupPermissions(@Nonnull String group, @Nonnull Set<String> permissions) {
        // Save to your database
    }

    @Override
    public void removeGroupPermissions(@Nonnull String group, @Nonnull Set<String> permissions) {
        // Remove from your database
    }

    @Override
    public Set<String> getGroupPermissions(@Nonnull String group) {
        // Query your database
        return Set.of();
    }

    // User-group membership methods
    @Override
    public void addUserToGroup(@Nonnull UUID uuid, @Nonnull String group) {
        // Save to your database
    }

    @Override
    public void removeUserFromGroup(@Nonnull UUID uuid, @Nonnull String group) {
        // Remove from your database
    }

    @Override
    public Set<String> getGroupsForUser(@Nonnull UUID uuid) {
        // Query your database
        return Set.of("Default");
    }
}
```

<Callout type="warning">
Hytale automatically assigns players to game-mode groups (like `Creative` or `Adventure`) using the first available provider.

If your provider throws an error when the group doesn't exist, the player will be disconnected! Always handle missing groups gracefully.
</Callout>

### Registrando seu `provider`

```java
public class MyPlugin extends JavaPlugin {

    @Override
    public void setup() {
        // Adiciona seu provider junto a outro por padrão
        PermissionsModule.get().addProvider(new DatabasePermissionProvider());
    }
}
```

O `PermissionsModule` agrega permissões de todos os _providers_ registrados. Isso significa que:

- As verificações consultam todos os _providers_;
- As operações de escrita (adicionar/remover) usam o **primeiro** _provider_.

### Substituindo o `provider` padrão

Se deseja ter controle total, pode remover o _provider_ padrão:

```java
public class MyPlugin extends JavaPlugin {

    @Override
    public void setup() {
        PermissionsModule perms = PermissionsModule.get();

        // Remove the default provider FIRST
        perms.removeProvider(perms.getFirstPermissionProvider());

        // Then add your provider
        perms.addProvider(new DatabasePermissionProvider());
    }
}
```

<Callout type="info">
Ao remover o provider padrão, o comando `/op self` deixará de funcionar porque ele verifica se houve manipulação de providers.
</Callout>

## Boas práticas

### Convenções de nomenclatura de permissões

Siga o padrão: `namespace.category.action`

```java
// Bom — hierarquia objetiva
"myplugin.admin.ban"
"myplugin.user.teleport.home"
"myplugin.vip.chat.color"

// Ruim — estrutura confusa
"myplugin_ban"
"teleport"
"vipChatColor"
```

### Use constantes para permissões

Defina permissões como constantes para evitar erros de digitação:

```java
public final class MyPermissions {
    public static final String ADMIN_BAN = "myplugin.admin.ban";
    public static final String ADMIN_KICK = "myplugin.admin.kick";
    public static final String USER_HOME = "myplugin.user.home";
    public static final String VIP_FLY = "myplugin.vip.fly";
}

// Uso
if (player.hasPermission(MyPermissions.ADMIN_BAN)) {
    // ...
}
```

### Evite excesso de permissões

Crie permissões apenas para ações que realmente precisam de controle de acesso. Nem tudo exige uma verificação de permissão.

### Segurança de threads

Ao criar um _provider_ personalizado, garanta que ele seja um _thread seguro_. As verificações de permissão podem ocorrer simultaneamente em múltiplas threads. Use `ReadWriteLock` ou `ConcurrentHashMap` nas suas estruturas de dados.

## Comandos no jogo

O Hytale fornece comandos integrados para gerenciar permissões.

### Comando `/op`

| Comando               | Descrição                                                                         |
| --------------------- | --------------------------------------------------------------------------------- |
| `/op self`            | Alterna seu próprio status de OP (modo um jogador ou com a bandeira `--allow-op`) |
| `/op add <player>`    | Adiciona um jogador ao grupo OP                                                   |
| `/op remove <player>` | Remove um jogador do grupo OP                                                     |

### Comando `/perm`

Use `/perm --help` no jogo para ver o uso completo. Operações comuns:

| Comando                                  | Descrição                          |
| ---------------------------------------- | ---------------------------------- |
| `/perm user list <uuid>`                 | Lista as permissões do usuário     |
| `/perm user add <uuid> <perm>`           | Adiciona uma permissão ao usuário  |
| `/perm user remove <uuid> <perm>`        | Remove uma permissão do usuário    |
| `/perm user group list <uuid>`           | Lista os grupos do usuário         |
| `/perm user group add <uuid> <group>`    | Adiciona o usuário a um grupo      |
| `/perm user group remove <uuid> <group>` | Remove o usuário de um grupo       |
| `/perm group list <group>`               | Lista as permissões do grupo       |
| `/perm group add <group> <perm>`         | Adiciona uma permissão ao grupo    |
| `/perm group remove <group> <perm>`      | Remove uma permissão do grupo      |
| `/perm test <perm>`                      | Testa se você possui uma permissão |

## Resumo

Você aprendeu a:

- **Verificar as permissões** usando `PermissionsModule.hasPermission()`;
- **Gerenciar as permissões de usuários** com `addUserPermission()` e `removeUserPermission()`;
- **Trabalhar com grupos** usando `addGroupPermission()`, `addUserToGroup()` etc;
- **Usar wildcards** como `*` e `myplugin.*` para permissões em massa;
- **Monitorar eventos** de mudanças de permissão;
- **Criar providers personalizados** para permissões baseadas em banco de dados.

Para sistemas de permissão mais avançados, com recursos como herança, expiração e contextos, estamos avaliando a expansão desse guia no futuro.
